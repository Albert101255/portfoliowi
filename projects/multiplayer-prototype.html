Project 05

Multiplayer Mini-Game Prototype

A networked Unreal Engine 5 obstacle course with C++ AI enemies

// Overview

Unreal Engine with C++ gameplay code is the standard in AAA game development, and this prototype was built specifically to move beyond Blueprint drag-and-drop and into the engine's actual programming layer. The result is a networked third-person obstacle course prototype: players navigate a gauntlet of physics obstacles and encounter basic enemy AI that patrols, detects the player, and gives chase.

The prototype supports 2-player local split-screen. Each player has a character controller written in C++, a health system, a checkpoint system, and a time-attack scoring screen at the end. Enemy AI is implemented via Unreal's Behavior Tree system, driven by a custom C++ BTTaskNode for the patrol path logic.

// What I Built
  • › C++ ACharacter subclass with full movement (walk, run, jump, crouch) and a health component written as a C++ UActorComponent.
  • › Three obstacle types: rotating platforms (Blueprint-driven physics), timed pressure plates (C++ trigger overlap events), and projectile launchers (C++ spawn actor with initial velocity).
  • › Basic enemy AI: an AAIController with a Behavior Tree containing patrol, detect-player, and chase states. The patrol path uses custom C++ UBTTaskNode_Patrol that reads waypoints from ATargetPoint actors.
  • › Local multiplayer: Unreal's built-in split-screen with separate APlayerController instances. Both players share the same level stream — tested stable at 120FPS in-editor.
  • › A simple UI (UMG widget) showing per-player health and a countdown timer, updated via C++ TBroadcast delegate.
// What I Learned

Unreal's C++ layer is not beginner-friendly — understanding the distinction between UObject, AActor, APawn, ACharacter, and when to use each was the first 30% of the project's learning curve. The engine's macro system (UPROPERTY, UFUNCTION, GENERATED_BODY()) felt like magic until reading the generated header files and understanding what the Unreal Header Tool actually produces.

The biggest insight: game feel is not code, it's numbers. The difference between "floaty jump" and "satisfying jump" was entirely in the JumpZVelocity and GravityScale values — no code changes required. Playtesting versus tweaking parameters was a session unto itself.

// The Challenge

Getting the enemy AI to feel "fair" — enemies that are too precise feel unfair; enemies with a delay feel dumb. Finding the sweet spot required careful tuning of the perception component's detection radius and sight angle.

// The Approach

Used Unreal's UAIPerceptionComponent with a sight sense configured to a 90-degree forward arc and 1200cm detection range, with a 0.3-second reaction delay before entering chase state. The reaction delay is implemented as a Blueprint delay node on the behavior tree branch — intentionally done in Blueprint to make it designer-adjustable without recompiling C++.
