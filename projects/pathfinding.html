Project 01

High-Performance Pathfinding Visualizer

Real-time shortest path finding with A* and Dijkstra's Algorithm

// Overview

Pathfinding is the invisible engine behind every GPS, every game NPC, and every logistics network — and this project makes it visible. The Pathfinding Visualizer is a desktop application built in C++ with the SFML graphics library, where users can construct custom grid maps with obstacles, define start and end points, and watch in real-time as the algorithm navigates the maze to find the shortest route.

The application supports two of the most fundamental graph traversal algorithms: A* (A-Star), which uses heuristics to intelligently prioritize paths, and Dijkstra's Algorithm, which exhaustively explores all nodes to guarantee the optimal solution. The real-time visualization makes abstract Computer Science theory tangible — users can watch the algorithm "think," observing how it expands outward from the source node, paints visited nodes, and backtracks to trace the final path.

// What I Built
  • › An interactive grid system rendered frame-by-frame using SFML, supporting click-to-place obstacles, drag-to-draw walls, and right-click-to-clear.
  • › A full A* implementation with Manhattan distance heuristic and configurable weight factors.
  • › Dijkstra's Algorithm implementation sharing the same node/graph architecture as A* for performance comparison.
  • › A step-by-step animation mode and an instant-solve mode with path length and nodes-explored counters displayed on screen.
  • › Clean, efficient memory management with zero dynamic allocation inside the hot rendering loop — all pre-allocated node pools.
// What I Learned

Writing this visualizer forced a deep understanding of priority queue behavior in C++ — specifically how std::priority_queue behaves versus a custom binary heap when processing thousands of nodes per frame. The difference in frame-time between naïve and optimized implementations was measurable and visible, which is the kind of hands-on performance insight that no textbook delivers.

The project also reinforced the importance of separating rendering logic from algorithm logic. Early drafts had the pathfinding loop directly mutating render state, causing flickering and race-like behavior. Refactoring into a clean state machine eliminated all visual artifacts and made the code dramatically more testable.

// The Challenge

Making the algorithm visualization run at a smooth 60 FPS while still being slow enough to watch — the algorithm itself runs in microseconds, so a deliberate animation scheduler with configurable playback speed had to be engineered from scratch.

// The Approach

Implemented a frame-budget animation queue: each "step" of the algorithm (one node expansion) is pushed to a queue, and the renderer pops and draws one step per configurable interval (10ms to 500ms, user-adjustable with a slider). The algorithm runs to completion first, then the visualization plays back at controlled speed.
