Project 04

Competitive Battle Calculator

A precision damage calculator built on type-logic and stat modifiers

// Overview

For competitive battling communities, damage calculators are the spreadsheets of strategy — every serious player uses one, and their accuracy is non-negotiable. This Battle Calculator lets users input two entities with customizable base stats (HP, Attack, Defense, Speed, Special Attack/Defense), select an "attack" with power and type, and instantly see the damage output — including minimum/maximum rolls, type effectiveness multipliers, and whether the attack is a guaranteed one-hit.

The project was built with a laser focus on two things: mathematical correctness and testability. Every formula is extracted into a pure function, and every pure function has a battery of unit tests with known expected outputs.

// What I Built
  • › A complete damage formula engine in C# covering: base damage, type effectiveness chart (18×18 types), STAB multiplier, critical hit probability, and random roll range.
  • › A full NUnit test suite: 47 unit tests covering normal damage cases, edge cases (0 damage type immunity), multi-hit moves, and critical hit scenarios. All 47 pass.
  • › A web frontend with stat input fields, move type selector (color-coded type badges), and an output panel showing expected damage, damage range, type effectiveness label, and a visual HP bar.
  • › Input validation preventing negative stats, stat totals above logical maximums, and zero-power moves.
// What I Learned

Unit testing stopped being an abstract best practice and became a concrete lifesaver on this project. The damage formula has six compounding multipliers — catching a bug in the type effectiveness lookup was only possible because the failing test output told me exactly which case broke. Without tests, that bug would have been invisible.

C#'s strong type system proved its value. Modeling types as enums rather than strings eliminated an entire class of potential bugs — the compiler would reject a typo in a type name, where a string comparison would silently fail.

// The Challenge

Representing the full type effectiveness chart efficiently — 18×18 = 324 relationships — without hard-coding 324 conditional statements.

// The Approach

Stored the chart as a 2D array indexed by two enums cast to int. The entire type chart loads once at startup. Lookups are O(1) array accesses. Adding a new type is one row+column addition, not a code rewrite.
